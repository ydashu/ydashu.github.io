邮件端口及加密方式

2025-09-02

587 vs 465 端口 + SSL/TLS/STARTTLS 综合对比表

项目 端口 587（Submission） 端口 465（SMTPS）
全称 Message Submission Agent (MSA) SMTP over SSL（旧称 SMTPS）
用途 邮件提交（用户发送邮件） 邮件发送（历史加密方式）
加密方式 STARTTLS（明文连接 → 协商升级为 TLS） 直接 SSL/TLS 加密连接（一建立就加密）
是否标准 ✅ IETF 官方推荐标准（RFC 6409） ⚠️ 曾被废弃，后用于兼容性支持
安全性 高（支持现代 TLS，防止降级） 高（若配置得当），但依赖历史实现
是否需要认证 ✅ 必须登录（用户名 + 密码/授权码） ✅ 通常也需要认证
防火墙穿透能力 ✅ 好（不被 ISP 封锁） ✅ 一般也开放
使用场景 现代邮件客户端、云服务（Gmail、AWS SES） 国内邮箱（腾讯企业邮、阿里邮箱）
推荐程度 ✅ 强烈推荐（新项目首选） ⚠️ 仅用于兼容旧系统

SSL、TLS、STARTTLS 术语解析表

术语 类型 含义 状态 说明
SSL
(Secure Sockets Layer)		 加密协议 早期安全协议，用于加密网络通信 ❌ 已淘汰
（SSLv2/v3 有严重漏洞）		 虽然常说“SSL证书”，但实际上是指 TLS 证书
TLS
(Transport Layer Security)		 加密协议 SSL 的继任者，现代加密标准 ✅ 当前主流协议
（TLS 1.2/1.3 最安全）		 所有 HTTPS、SMTP、IMAP 安全通信的基础
STARTTLS 协议命令 用于将明文连接“升级”为 TLS 加密连接 ✅ 推荐使用机制 工作在应用层，常见于端口 25、587、143、110

💡 比喻：

  • SSL = 老旧防盗门（已过时）
  • TLS = 新式智能锁（现代标准）
  • STARTTLS = “我现在要上锁了” 的指令，触发加密

关键关系说明

关系描述 说明
STARTTLS 使用的不是 SSL 虽然名字叫 STARTTLS，但它实际启用的是 TLS 协议，不是 SSL！
465 端口 ≠ 不安全 只要使用 TLS 1.2+，即使走 465，也是安全的。安全性取决于加密算法，不是端口号
587 端口 是现代标准 IETF 明确推荐 587 为“邮件提交”端口，是云服务、API 集成的首选
“SSL 加密”是口语说法 实际上所有现代服务都使用 TLS，但习惯称“开启SSL”

主流邮箱服务对比（支持情况）

服务商 推荐端口 支持 587 支持 465 推荐加密方式
腾讯企业邮 (exmail.qq.com) ⚠️ 465（文档推荐） ✅ 支持 ✅ 支持 SSL (465) 或 STARTTLS (587)
阿里企业邮箱 ⚠️ 465 ✅ ✅ SSL
Gmail / Google Workspace ✅ 587 ✅ ❌（不支持传统 465） STARTTLS
Outlook.com / Hotmail ✅ 587 ✅ ❌（不支持） STARTTLS
Amazon SES ✅ 587 ✅ ❌（已停用） STARTTLS
SendGrid / Mailgun ✅ 587 ✅ ❌ STARTTLS

✅ 国际趋势：全面转向 587 + STARTTLS
⚠️ 国内现状：465 + SSL 仍占主流，主要是历史和用户习惯原因

最佳实践建议总结

场景 推荐配置
🆕 新项目开发（网站、API、云服务） ✅ 使用 587 + STARTTLS + TLS 1.2+
💼 企业内部系统、兼容老客户端 可使用 465 + SSL/TLS（需确保加密强度）
🔐 安全性要求高 强制启用加密、禁用 SSLv3 及弱算法
📧 客户端配置（如 Outlook、Thunderbird） 优先选择“STARTTLS on port 587”
📱 手机邮箱 App 配置 根据服务商提示选择，腾讯邮可用 465

总结一句话：

✅ 技术先进性排序：587 + STARTTLS + TLS > 465 + TLS > 465 + SSL
🔐 安全的关键是加密协议（TLS）和证书，而不是端口号。
🌍 国际趋势用 587，国内习惯用 465 —— 了解差异，灵活选择。
